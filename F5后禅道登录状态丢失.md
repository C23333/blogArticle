# F5后禅道登录状态丢失

F5 接上以后，禅道页面能打开，登录也能成功。

但是点几下菜单，又像没登录一样跳回去了。这个问题一开始看很像禅道自己的登录问题，后来发现是多节点后 session 没统一，F5 又把请求打到了不同应用机。

## 现场结构

```text
F5 VIP -> app01:80
       -> app02:80

app01/app02 -> SQL Server Listener
app01/app02 -> Redis Sentinel / Redis master
```

## 先确认是不是打到不同节点

在 app01/app02 的 Nginx access log 里加一下 upstream 和 hostname 更好查。

临时也可以在页面放一个节点标识测试文件：

```bash
echo app01 > /data/www/zentao/www/node.txt
```

app02：

```bash
echo app02 > /data/www/zentao/www/node.txt
```

通过 F5 访问：

```bash
curl http://zentao.example.com/node.txt
```

多刷几次，如果一会儿 app01，一会儿 app02，说明负载是生效的。

## session 如果还在本机文件

PHP 默认 session 可能在本机文件：

```bash
php -i | grep session.save_path
php -i | grep session.save_handler
```

如果 app01 登录时 session 文件写在 app01，本次请求到了 app02，app02 找不到这个 session，自然就像没登录。

## 方案一：F5 做会话保持

可以让同一个用户尽量落到同一台应用机，比如 cookie persistence 或 source address affinity。

但这只能缓解，不是我最喜欢的方式。因为节点发布、下线、客户端出口 IP 变化时还是会有影响。

## 方案二：session 放 Redis

更适合多节点。

PHP 安装 redis 扩展：

```bash
yum install -y php-pecl-redis
systemctl restart php-fpm
php -m | grep redis
```

配置 PHP session 示例：

```ini
session.save_handler = redis
session.save_path = "tcp://10.10.30.11:6379?auth=<REDIS_PASSWORD>"
```

如果应用支持 Sentinel，优先让应用从 Sentinel 拿 master。否则至少要考虑 Redis 切主后配置怎么更新，不要写死老 master。

重启：

```bash
systemctl restart php-fpm
```

测试 session 文件是否还在本地增长：

```bash
ls -lt /var/lib/php/session | head
```

Redis 里看 key：

```bash
redis-cli -h 10.10.30.11 -a '<REDIS_PASSWORD>' keys '*'
```

生产不要随便 `keys *`，这里只是测试环境验证。生产看数量可以用 scan。

## 附件也要统一

session 只是一个坑，附件是另一个坑。

如果 app01 上传附件到本机 `/data/upload/zentao`，下一次请求到 app02，app02 本地没有这个文件，也会报找不到附件。

至少要做其中一种：

```text
1. 共享存储挂到两台同一路径
2. 对象存储
3. 发布时限制附件只落一处，然后应用统一读
```

当时先按共享目录处理，两台路径保持一致：

```bash
/data/upload/zentao
```

权限：

```bash
chown -R nginx:nginx /data/upload/zentao
```

## F5 健康检查别只看 80 端口

只检查 80 端口，Nginx 活着就绿了，但 PHP-FPM、数据库、Redis 都可能有问题。

建议做一个简单健康检查地址，比如：

```text
/health.php
```

里面至少检查：

```text
1. PHP 正常执行
2. SQL Server 能 SELECT 1
3. Redis 能 PING
```

F5 monitor 指向这个地址，比只 TCP check 靠谱。

## HTTPS 变 HTTP

F5 如果做 HTTPS 卸载，后端走 HTTP，应用里可能拿到的是 http，导致跳转地址变成 HTTP。

要看 F5 是否传：

```text
X-Forwarded-Proto: https
X-Forwarded-For: client_ip
```

Nginx 里也要把头传给 PHP：

```nginx
fastcgi_param HTTP_X_FORWARDED_PROTO $http_x_forwarded_proto;
fastcgi_param HTTP_X_FORWARDED_FOR   $http_x_forwarded_for;
```

如果后端应用不识别这个头，还要在应用配置里指定外部访问地址。

## 真实 IP

F5 后面看到的 IP 可能全是 F5 地址。这个要让 F5 插入 `X-Forwarded-For`，后端日志格式也要记录。

Nginx log_format 示例：

```nginx
log_format main '$remote_addr $http_x_forwarded_for [$time_local] "$request" '
                '$status $body_bytes_sent "$http_referer" "$http_user_agent"';
```

## 最后查问题的顺序

```text
1. 多刷 node.txt，看是不是多节点
2. 看 PHP session handler
3. 看 Redis session 是否生效
4. 看附件目录是否共享
5. 看 F5 persistence
6. 看健康检查是不是只检查端口
7. 看 X-Forwarded-Proto / X-Forwarded-For
```

这次登录状态丢失，根因不是 F5 坏了，是应用多节点后 session 还按单机思路在跑。
